# Ascend Forge UI V3 Operational Audit

Date: 2026-06-11

## Scope

This audit maps the current Ascend Forge UI/runtime surface before implementation. The goal is operational connectivity, not visual redesign: every page should read from real state, call real endpoints, react to real events, and show unsupported states honestly.

## System Map

Ascend Forge currently has a stronger backend than the UI exposes.

- Backend router: `backend/routes/forge.js`
  - Projects, plans, runs, streamed runs, approvals, verification, apply, patches, replay, backlog, cycles, memory, learning, distillation, training, cognitive/memory graph, and Forge metrics.
- Backend persistence: `backend/services/forge_store.js`
  - SQLite-first store with JSON fallback for runs, actions, patches, audit events, metrics, and learning/memory support through the Forge learning proxy.
- Backend route registry: `backend/routes/index.js`
  - Lists canonical Forge routes plus legacy `/api/forge/queue`, `/submit`, `/approve`, `/reject`, `/rollback`, `/status` routes in `server.js`.
- Frontend API: `frontend/src/api/client.js`
  - Contains a large `api.forge` surface but is missing a runtime aggregate endpoint, run pause/resume/cancel, diagnostics, reports, run audit, and run report helpers.
- Frontend real-time layer: `frontend/src/hooks/useWebSocket.js`
  - Handles system, turn, approval, workflow, task, memory, cognitive, agent, economy, and security events.
  - Forge events are not first-class and mostly fall into the generic event feed.
- Legacy app store: `frontend/src/store/appStore.js`
  - Owns `forgeQueue` and `runtimeState.ascend_forge`, which duplicates newer Forge run concepts but does not model the full lifecycle.
- Forge UI page: `frontend/src/components/pages/AscendForgePage.jsx`
  - Provides the primary Forge workspace but keeps important run state locally and still uses raw `fetch`, `JPOST`, local toggles, fake agents, and decorative metric data.

## Pages And Current Connectivity

### Command Center / Nexus OS Dashboard

File: `frontend/src/components/NexusOSDashboard.jsx`

- Real inputs: task store, agent store, system store, event feed, security/economy stores.
- Missing Forge connectivity: no canonical Forge runtime snapshot, no active run summary, no linked approvals/validation/report/memory lifecycle.
- Risk: the landing page cannot answer what Forge is doing without navigating into separate pages.

### Ascend Forge

File: `frontend/src/components/pages/AscendForgePage.jsx`

- Real inputs:
  - `/api/forge/projects`
  - `/api/forge/sessions`
  - `/api/forge/runs/stream`
  - `/api/forge/agentic-run`
  - `/api/forge/runs`
  - `/api/forge/files/tree`
  - `/api/forge/files/read`
  - `/api/forge/actions/:id/approve`
  - `/api/forge/actions/:id/reject`
- Local-only or fake inputs:
  - `runState` controls LIVE/PAUSED/IDLE locally.
  - Pause/resume button only toggles local state.
  - Agent rows are derived from `runState`, not backend telemetry.
  - Sparkline data is generated by `useRollingSpark()` and presented like operational metrics.
  - Some success toasts are optimistic and do not refresh canonical backend state.
- Missing:
  - Canonical active run/project/cycle selection.
  - Runtime snapshot hydration.
  - First-class Forge WebSocket updates.
  - Honest unsupported states for pause/resume/cancel.

## Backend API Inventory

Existing Forge route families:

- Projects: list, create, import, get, delete.
- Sessions: create, message, streamed message.
- Runs: create, stream, list, get, approve, verify, apply, transcript, pending approvals, approve action, reject action, continue.
- Evidence: patches, replay, metrics.
- Backlog/autopilot/cycles.
- Learning/memory: project memory, learning summary, lessons, distillation, promote lesson, datasets, preference pairs, evaluation cases, skill proposals.
- Cognitive: memory graph, context packets, advisory events/metrics, cognitive events.
- Training/model versions.

Missing or incomplete for an operational UI:

- `GET /api/forge/runtime`
- `GET /api/forge/diagnostics`
- `GET /api/forge/reports`
- `GET /api/forge/runs/:id/audit`
- `GET /api/forge/runs/:id/report`
- `POST /api/forge/runs/:id/pause`
- `POST /api/forge/runs/:id/resume`
- `POST /api/forge/runs/:id/cancel`

## Event Stream Audit

Current `useWebSocket.js` routes many event prefixes, but Forge is not a canonical event domain.

Needed first-class events:

- `forge:runtime_snapshot`
- `forge:project_updated`
- `forge:run_created`
- `forge:run_updated`
- `forge:action_updated`
- `forge:approval_required`
- `forge:approval_decided`
- `forge:validation_started`
- `forge:validation_completed`
- `forge:report_generated`
- `forge:memory_candidate_created`
- `forge:diagnostic`

Fallback requirement:

- One centralized Forge polling loop, at most every 10 seconds, only when WebSocket is disconnected/stale or a mutation needs forced refresh.
- No page-specific Forge polling for operational truth after migration.

## State Duplication And Fake Data

Duplicated or disconnected state:

- `AscendForgePage` local `runState`.
- `AscendForgePage` local `activeRun`, `actions`, and `messages` as the only page truth after streamed runs.
- `appStore.forgeQueue` for legacy queue state.
- Page-specific fetch and refresh loops in Forge, Proof, Approval, Workflow, and Memory surfaces.

Fake/demo operational signals:

- `AscendForgePage` fake planner/coder agents derived from `runState`.
- Random sparkline data from `useRollingSpark()`.
- Local pause/resume state toggle with no backend effect.
- Success toasts that do not always reflect returned backend state.

## Risk Areas

- Existing worktree is dirty; unrelated changes must not be reverted.
- `backend/routes/forge.js` is large and owns many route families. Additive endpoints are safer than route rewrites.
- Streaming Forge runs may not currently have a tracked AbortController. Cancel must either cancel a tracked run or return an honest unsupported/conflict response.
- Approval gates must remain intact. Verification/apply actions that require owner approval must still enforce that requirement.
- The UI must not fabricate run, validation, memory, or report state when an endpoint is missing or unavailable.

## Implementation Checklist

1. Add backend runtime snapshot endpoint.
2. Add diagnostics, reports, run audit, run report, and honest run controls.
3. Broadcast Forge events from Forge mutations using the existing broadcaster.
4. Add `api.forge` helpers for all new endpoints.
5. Add canonical `forgeStore` with runtime hydration, normalized maps, relationships, unsupported actions, freshness, and polling fallback.
6. Route Forge-related WebSocket events into `forgeStore`.
7. Connect Ascend Forge to backend run controls and canonical runtime state.
8. Connect Approval Inbox refresh/relationships to canonical Forge state.
9. Connect Proof Center validation retry/evidence to Forge runtime.
10. Connect Command Center summary to active Forge lifecycle state.
11. Surface Forge lessons/memory candidates from the canonical runtime.
12. Mark unsupported actions visibly instead of faking success.
13. Document implementation and run focused backend/frontend verification.
