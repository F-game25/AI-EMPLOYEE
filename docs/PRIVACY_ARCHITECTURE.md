## 0. Executive Summary

AI-EMPLOYEE is a hybrid AI Operating System running in two deployment shapes (single-user desktop and multi-tenant SaaS). This document defines the architecture that enforces a single non-negotiable invariant:

> **No developer, operator, or super-admin can read user content under normal operation. All reads of user-generated data require explicit, time-boxed, user-consented break-glass procedure, and every such read is logged in an append-only hash-chained audit log.**

Developers retain full operational visibility (uptime, latency, error rates, token usage, cost) via a separate analytics database that contains zero PII and zero plaintext content.

This is achieved through five reinforcing controls:

1. **Per-tenant per-field envelope encryption** (DEKs wrapped by tenant KEKs wrapped by a root KEK in HashiCorp Vault Transit).
2. **A single sanitization pipeline** at the boundary of every logger, tracer, and external sink.
3. **Operational/analytics database split** with no foreign keys between them and no shared service accounts.
4. **OPA-evaluated RBAC + ABAC** where the default policy answer for `read_user_content` is `deny` for every role.
5. **Break-glass with mandatory user consent, time-boxing, notification, and forensic audit**.

The current codebase already contains the right scaffolding (`runtime/core/tenancy.py`, `runtime/core/tenant_middleware.py`, `backend/tenancy.js`, `backend/security/secrets.js`, `runtime/core/audit_engine.py`) but is currently file-based, lacks field-level encryption, and has a shared Chroma collection across tenants. The migration plan in Section 10 takes the system from current to target without downtime.

## 1. Key Hierarchy: Per-Tenant + Per-Field DEKs

### 1.1 Three-tier envelope encryption

```
                    ┌──────────────────────────────────────────┐
                    │   ROOT KEK (Vault Transit, never leaves) │
                    │   key: ai-employee/transit/root-kek       │
                    │   alg: aes256-gcm96                       │
                    │   rotation: quarterly                     │
                    └──────────────────┬───────────────────────┘
                                       │ wraps
                ┌──────────────────────┼──────────────────────┐
                ▼                      ▼                      ▼
        ┌───────────────┐      ┌───────────────┐      ┌───────────────┐
        │ Tenant A KEK  │      │ Tenant B KEK  │      │ Tenant C KEK  │
        │ (Transit key) │      │ (Transit key) │      │ (Transit key) │
        │ name: kek-A   │      │ name: kek-B   │      │ name: kek-C   │
        └───────┬───────┘      └───────────────┘      └───────────────┘
                │ wraps
    ┌───────────┼────────────┬──────────────┬──────────────┐
    ▼           ▼            ▼              ▼              ▼
┌────────┐ ┌────────┐  ┌──────────┐  ┌──────────┐  ┌──────────────┐
│ DEK    │ │ DEK    │  │ DEK      │  │ DEK      │  │ DEK          │
│ chat_  │ │ memory │  │ vault_   │  │ pending_ │  │ audit_       │
│ msgs   │ │ entries│  │ notes    │  │ review   │  │ payload      │
│ AES-256│ │ AES-256│  │ AES-256  │  │ AES-256  │  │ AES-256      │
│ -GCM   │ │ -GCM   │  │ -GCM     │  │ -GCM     │  │ -GCM         │
└────────┘ └────────┘  └──────────┘  └──────────┘  └──────────────┘
```

**Root KEK** lives only inside Vault Transit. It is never exportable. All wrap/unwrap of tenant KEKs happens via Vault's `transit/encrypt/root-kek` and `transit/decrypt/root-kek` endpoints. The application never sees the root key material.

**Tenant KEKs** are also Vault Transit keys (one per tenant, named `kek-<tenant_id>`). They are created lazily at tenant provisioning. They are used to wrap field-level DEKs.

**Field-level DEKs** are 32-byte AES-256-GCM keys generated by the application, immediately wrapped by the tenant KEK via `transit/encrypt/kek-<tenant_id>`, and stored as ciphertext in the `field_keys` table. They are unwrapped on demand (cached in memory for the duration of a request, never persisted decrypted).

### 1.2 Fields that get field-level encryption (FLE)

| Field                                       | Table                  | Rationale                              |
|---------------------------------------------|------------------------|----------------------------------------|
| Vault note body                             | `vault_notes.body`     | Direct user knowledge content          |
| Vault frontmatter (sources, tags)           | `vault_notes.frontmatter` | May leak identity/affiliation      |
| Chat message content                        | `chat_messages.content`| Direct user input + LLM output         |
| Memory entry text                           | `memory_entries.content` | Long-term user-bound semantic data   |
| Memory metadata                             | `memory_entries.metadata_encrypted` | May include source URLs/names |
| Pending-review claim                        | `pending_review.claim` | Research output not yet user-approved  |
| Pending-review verification                 | `pending_review.verification` | LLM reasoning over user data    |
| Agent prompt (rendered, with context)       | `agent_invocations.prompt` | Contains user data in interpolation |
| Agent response                              | `agent_invocations.response` | LLM output bound to user        |
| Document upload bytes                       | `documents.blob`       | User-supplied files                    |
| Audit log payload                           | `audit_log.payload_encrypted` | Action details may carry IDs    |
| User email                                  | `users.email`          | PII per GDPR                           |
| SSO subject                                 | `users.sso_subject`    | External-IdP identity                  |
| Topic intelligence labels                   | `topic_intelligence.label`, `.scope` | Reflects user interests   |
| Learning session result summary             | `learning_sessions.result_summary` | LLM-generated content        |

### 1.3 Algorithms

- **Symmetric encryption:** AES-256-GCM with 96-bit IVs. Authenticated, no padding oracles, fast on commodity CPU with AES-NI. Format on disk: `b'\x01' || iv(12) || ciphertext || tag(16) || aad_hash(32)`. Leading version byte allows future migration.
- **Tenant KEK wrap of DEK:** Vault Transit `aes256-gcm96`. Vault stores the key version internally and emits ciphertext blobs as `vault:v<n>:<base64>`.
- **Password hashing:** Argon2id (m=64 MiB, t=3, p=4). Not used as KEK; KEKs come from Vault, not from passwords. Password hashes are stored plaintext alongside ciphertext columns and protect login only.
- **Search over encrypted fields:** blind-indexed via HMAC-SHA256 of normalized lowercase value, keyed by a per-tenant per-field HMAC key (separate from encryption DEK). Stored in `*_search_hash` columns. Supports equality search; no range/prefix.
- **Hash chain in audit log:** SHA-256, `current_hash = SHA256(prev_hash || canonical_row_json)`.

### 1.4 Key rotation strategy

- **Root KEK:** quarterly. Use Vault `transit/keys/root-kek/rotate`. Old versions retained for decryption only (`min_decryption_version` floats forward over years, not weeks). No application-level work needed — Vault handles version transparently in the ciphertext header.
- **Tenant KEK:** on tenant lifecycle events (admin reset, suspected compromise, tenant moves to higher security tier). Same Vault rotate flow.
- **Field DEKs:** lazy-rotated on first write after a configured age (default 365d). New rows use the new DEK; old rows stay readable via the old DEK (still wrapped under the current tenant KEK).
- **Rewrap-on-rotation algorithm:** When a tenant KEK rotates, the application **does not decrypt user content**. It only reads each wrapped DEK ciphertext, calls `transit/rewrap/kek-<tenant_id>`, and writes the new wrapped DEK back. Vault performs the unwrap-with-old-version + wrap-with-new-version inside its own boundary. Application code never sees plaintext DEKs during rotation.

```python
# Pseudocode for tenant KEK rotation
for wrapped_dek in db.fetch_all("SELECT id, wrapped_dek FROM field_keys WHERE tenant_id=%s", tenant_id):
    new_wrapped = vault.transit.rewrap(key_name=f"kek-{tenant_id}", ciphertext=wrapped_dek)
    db.execute("UPDATE field_keys SET wrapped_dek=%s, kek_version=%s WHERE id=%s",
               new_wrapped, current_kek_version, wrapped_dek.id)
```

### 1.5 Why Vault Transit (not Vault KV v2)

Transit is a *cryptographic service*: plaintext keys live inside Vault forever, and the application calls `encrypt`/`decrypt`/`rewrap`/`sign`/`verify` over HTTPS. KV v2 stores arbitrary secrets and *returns them in plaintext* to the caller. KEKs must never appear in application memory, so KV is wrong for KEKs. KV v2 is still used for service-account credentials, third-party API keys, and signing keys that the app legitimately needs in memory.

## 2. HashiCorp Vault Deployment Topology

### 2.1 Desktop mode

The user runs AI-EMPLOYEE locally. We cannot assume an external Vault cluster. Design:

- **External Vault binary** (`vault server -dev-tls=false -config=~/.ai-employee/vault.hcl`) launched by `start.sh` as a child process. Backend: integrated file storage with single-node raft (`storage "raft" { path = "~/.ai-employee/vault-data" }`).
- **Auto-unseal via OS keyring**, not the deprecated `-dev` mode:
  - Linux: libsecret (DBus secret-service)
  - macOS: Keychain
  - Windows: DPAPI
  - Implementation: a small Go shim (or Python `keyring` package) reads/writes the unseal key. Vault is initialized once; the unseal key is wrapped by the OS keyring at init and unwrapped on each app start.
- **First-boot wizard** (`start.sh` → `runtime/setup/first_boot.py`) generates the unseal key, writes it to the OS keyring under service name `ai-employee.vault.unseal`, captures a user passphrase, derives a passphrase-KEK with Argon2id, and wraps the *root token* with the passphrase-KEK. The wrapped root token sits in `~/.ai-employee/vault-root.enc`. Daily operation never uses the root token; the wizard immediately provisions an `app` AppRole and stores the SecretID via Vault's response wrapping.
- **Service account tokens** issued to the Node and Python processes are AppRole tokens with TTL 15 minutes, periodic renewal, policy scoped to `transit/encrypt/kek-*`, `transit/decrypt/kek-*`, `transit/rewrap/kek-*`, and `kv/data/app/*`.

### 2.2 SaaS mode

- **Vault cluster:** 3 nodes minimum behind an internal load balancer (`vault.internal`). Raft integrated storage. Auto-unseal via cloud KMS (AWS KMS `awskms` seal, GCP `gcpckms`, or Azure Key Vault — environment-controlled).
- **Performance standby** enabled (Enterprise feature) or read-only replicas via Raft read-after-write semantics (OSS).
- **Network policy:** Vault is reachable *only* from the application VPC. Firewall rule: deny all inbound except port 8200 from `app-tier` security group. Vault itself has no egress except KMS for auto-unseal.
- **Audit device:** Vault's own audit log shipped to a write-only S3 bucket with object lock (compliance mode, 7-year retention). This is separate from the application audit log in §3.
- **Backup:** Raft snapshots every 4h to encrypted cold storage. See §7.

### 2.3 Token lifecycle for application services

| Token type        | TTL   | Renewable | Scope                                       |
|-------------------|-------|-----------|---------------------------------------------|
| Root              | n/a   | n/a       | Sealed in OS keyring (desktop) or quorum-shared (SaaS). Only used for break-glass admin ops. |
| AppRole (app)     | 15min | yes       | `transit/{encrypt,decrypt,rewrap}/kek-*`, `kv/data/app/*` |
| AppRole (audit)   | 15min | yes       | `transit/encrypt/audit-key` only (write-only audit signing) |
| Per-tenant token  | 5min  | no        | Scoped to `kek-<tenant_id>` only — used for cross-tenant safety check |

Never put a Vault token in `.env`. Tokens are fetched at process start via AppRole login (`role_id` baked into config, `secret_id` delivered via response wrapping by `start.sh`).

## 3. Postgres Schema with FLE

**Database split:** two physical databases.

1. **`aiemp_ops`** — operational data, FLE applied, accessed only by application service accounts.
2. **`aiemp_analytics`** — telemetry, no PII, accessed by developer dashboards.

No service account has rights to both.

### 3.2 Application-level FLE API

We use **application-level** encryption (not pgcrypto) because pgcrypto requires the DB to hold key material, which violates our "keys live in Vault only" rule.

```python
# runtime/core/crypto/field.py
class FieldCrypto:
    def __init__(self, vault: VaultTransitClient, tenant_id: UUID, field_label: str):
        self.vault = vault
        self.tenant_id = tenant_id
        self.field_label = field_label

    def encrypt(self, plaintext: bytes, aad: bytes = b"") -> bytes:
        dek = self._get_or_create_dek()      # cached for request scope
        iv = os.urandom(12)
        ct, tag = AES_GCM(dek).encrypt(iv, plaintext, aad)
        return b"\x01" + iv + ct + tag

    def decrypt(self, blob: bytes, aad: bytes = b"") -> bytes:
        assert blob[0] == 0x01
        iv, body = blob[1:13], blob[13:]
        ct, tag = body[:-16], body[-16:]
        dek = self._get_dek_for_decrypt()
        return AES_GCM(dek).decrypt(iv, ct, tag, aad)
```

AAD always includes `tenant_id || field_label || row_id` so a ciphertext copy-pasted into another row or another tenant fails authentication. This is the cryptographic enforcement of tenant isolation, complementary to the application-level check in `runtime/core/tenancy.py`.

## 4. Sanitization / Redaction Pipeline

### 4.1 Single chokepoint per runtime

- **Python:** `runtime/core/log_sanitizer.py` — wraps the standard `logging` module via a `Filter`. Every Python logger goes through `logging.getLogger().addFilter(SanitizerFilter())` at process start. Direct `print()` is banned (lint rule: `flake8-print`).
- **Node:** `backend/middleware/sanitize.js` — wraps `winston` (or whatever logger) via a format transform. Direct `console.log` is banned (eslint rule: `no-console`). The existing `backend/security/secrets.js` is extended to register the sanitizer.

### 4.2 Pattern catalog

| Pattern              | Regex / detector                                      | Replacement              |
|----------------------|-------------------------------------------------------|--------------------------|
| JWT                  | `eyJ[A-Za-z0-9_-]+\.[A-Za-z0-9_-]+\.[A-Za-z0-9_-]+`   | `<REDACTED:jwt>`         |
| OpenAI key           | `sk-[A-Za-z0-9]{20,}`                                 | `<REDACTED:openai_key>`  |
| Anthropic key        | `sk-ant-[A-Za-z0-9_-]{20,}`                           | `<REDACTED:anthropic_key>` |
| GitHub PAT           | `gh[pousr]_[A-Za-z0-9]{36}`                           | `<REDACTED:github_pat>`  |
| AWS access key       | `AKIA[0-9A-Z]{16}`                                    | `<REDACTED:aws_key>`     |
| Email                | RFC-5322-simplified                                   | `<REDACTED:email>`       |
| Phone (intl)         | `\+?\d[\d\s\-\(\)]{8,}\d`                             | `<REDACTED:phone>`       |
| Credit card          | 13–19 digits passing Luhn                             | `<REDACTED:card>`        |
| SSN (US)             | `\d{3}-\d{2}-\d{4}`                                   | `<REDACTED:ssn>`         |
| Vault note ID        | `note:[a-f0-9-]{36}`                                  | `<REDACTED:vault_ref>`   |
| Prompt body marker   | sentinel `<<<PROMPT_START>>>...<<<PROMPT_END>>>`      | `<REDACTED:prompt>`      |
| Base64 blob > 256B   | `[A-Za-z0-9+/=]{256,}`                                | `<REDACTED:blob>`        |

The sanitizer runs **before** the log line is serialized. Order of operations: (1) match all patterns, (2) replace with typed tokens, (3) if any match found, also emit `security.pii_detected_in_log` to the event bus with `{route, level, pattern_type, count}` (never the matched value), (4) write the redacted line.

### 4.3 Test suite

`tests/test_log_sanitizer.py` and `tests/test_sanitize_node.test.js` cover:
- single match per pattern
- multiple matches per line
- multi-line payload spanning the buffer
- base64-encoded JWT (double-encoded secrets)
- ANSI-escape-injection attempt
- structured log records with nested PII in dict values
- regression cases collected from past incidents (golden file)

CI fails if a `console.log`, `print(`, or `logger.x(<unsanitized>)` is committed.

### 4.4 Dashboard signal

`security.pii_detected_in_log` events surface in the developer dashboard as `Sensitive payload blocked` with count + route, never the value. This both protects the user and helps engineers find the bug that almost leaked.

## 5. Telemetry / Observability Separation

### 5.1 Two physical databases

```
┌───────────────────────────────┐      ┌───────────────────────────────┐
│   aiemp_ops (Postgres)        │      │   aiemp_analytics (Postgres)  │
│   - users, chat_messages, ... │      │   - request_metrics            │
│   - FLE on content columns    │      │   - error_events               │
│   - service account: app_rw   │      │   - cost_rollups               │
│   - service account: app_ro   │      │   - NO foreign keys to ops     │
│                               │      │   - service account: analytics │
└───────────────────────────────┘      └───────────────────────────────┘
        ▲                                       ▲
        │ only application service              │ only developer dashboards
        │                                       │
        └───────────── NO SHARED ACCOUNT ───────┘
```

### 5.2 Metric pipeline

Every HTTP request and every agent invocation passes through `runtime/core/observability/metrics_emitter.py`:

```python
metrics_emitter.emit({
    "request_id_hash": hmac(request_id),
    "tenant_id_hash":  hmac(tenant_id),
    "route":           route_template,    # parameterized, never path params
    "status":          status_code,
    "latency_ms":      latency_ms,
    "model":           model_name,
    "tokens_in":       tokens_in,
    "tokens_out":      tokens_out,
    "cost_cents":      cost_cents,
    "error_type":      error_class_name,  # never the message
    "ts":              now(),
})
```

The emitter buffers in memory and bulk-inserts to `aiemp_analytics.request_metrics` every 5s (or when the buffer hits 500 rows, whichever first). On crash, the buffer is lost — this is acceptable because analytics is best-effort, not the source of truth.

### 5.3 What is NEVER in analytics

- User IDs (only HMACs)
- Tenant names
- Prompt text
- Response text
- Error messages (only class names)
- File paths containing user content
- Stack traces (only an `error_type`)

### 5.4 Tracing

OpenTelemetry is enabled but spans go through `runtime/core/observability/span_scrubber.py` before export. Allowed attributes: `route`, `method`, `status`, `tenant_id_hash`, `request_id_hash`, `model`, `tokens`. Anything else is dropped. Span events that carry exception details are replaced with the exception class name.

## 6. RBAC + ABAC + Break-Glass

### 6.1 Role catalog

| Role            | Default capabilities                                            |
|-----------------|-----------------------------------------------------------------|
| `super_admin`   | Infra ops, tenant lifecycle, OPA policy edits. NOT content.     |
| `tenant_admin`  | Tenant-scope user mgmt, billing, settings. NOT content.         |
| `operator`      | Trigger deploys, view analytics DB, run runbooks.               |
| `analyst`       | Read-only analytics DB.                                         |
| `viewer`        | Read-only dashboard, tenant-scoped non-content (counts, status).|
| `support`       | Tenant-scope diagnostics: error class, latency. NOT content.    |
| `user`          | Owns content; full read/write within own tenant scope.          |

Critically: **no role grants `read_user_content` by default**, including `super_admin`. That capability is only obtainable via break-glass.

### 6.2 OPA policy evaluation

OPA runs as a sidecar process (`runtime/security/opa/`). The Python `runtime/security/opa_client.py` calls `POST /v1/data/aiemp/authz/allow` with:

```json
{
  "input": {
    "actor":    {"id": "...", "role": "operator", "tenant_id": "t1", "break_glass": false},
    "resource": {"type": "chat_message", "tenant_id": "t1", "id": "..."},
    "action":   "read_content",
    "context":  {"route": "/api/chat/messages/:id", "time": "...", "ip": "..."}
  }
}
```

Example Rego policy:

```rego
package aiemp.authz

default allow = false

# Self-ownership
allow {
  input.action == "read_content"
  input.actor.role == "user"
  input.actor.tenant_id == input.resource.tenant_id
  input.actor.id == data.resource_owner[input.resource.id]
}

# Break-glass content read
allow {
  input.action == "read_content"
  input.actor.break_glass == true
  token := data.break_glass_tokens[input.actor.id]
  time.now_ns() < token.expires_at_ns
  token.tenant_id == input.resource.tenant_id
}

# Admins NEVER pass this check without break-glass.
```

### 6.3 Break-glass procedure

```
1. Admin clicks "Request user content access" on a user record.
2. Modal requires:
   - Reason (> 50 chars, free text)
   - Time-box (15min | 1h | 4h max)
   - Scope (specific resource | tenant-wide)
3. POST /api/admin/break-glass/request
   → creates row in break_glass_requests (status='pending_user_consent')
   → fires in-app notification + email to affected user
   → fires audit_log entry (action='break_glass.requested')
4. User dashboard shows banner: "Admin <name> requests access to your <scope> for <reason>. Approve / Deny."
   - Approve: status → 'approved', token issued
   - Deny: status → 'denied', notification to admin, audit entry
   - No response in 24h: per-tenant policy decides:
       a) auto-deny (default for HIPAA tenants)
       b) auto-approve with elevated audit (default for regular tenants)
5. Token issued: JWT with `break_glass=true`, `bg_token_id=<uuid>`, `exp=<time-box>`, signed with separate break-glass signing key.
6. Every API call with break_glass=true:
   - Re-validates expiry
   - Emits audit_log entry (action='break_glass.read', resource_id_hash, payload_enc)
   - Pushes a real-time event to the affected user ("Admin <name> is currently viewing <resource>")
7. Token expiry:
   - Hard expiry at exp claim
   - Soft revocation if admin clicks "End session"
   - User can revoke at any time from dashboard
8. Daily summary:
   - Sent to security@<tenant> and operator@<deployment>
   - Lists all break-glass sessions, durations, resources accessed (hashed)
```

Files:
- `runtime/security/break_glass.py` — request lifecycle, token mint, revoke
- `runtime/security/break_glass_audit.py` — every read records an audit row
- `frontend/src/components/admin/BreakGlassPanel.jsx` — admin UI to request
- `frontend/src/components/user/BreakGlassConsentBanner.jsx` — user consent UX
- `frontend/src/components/user/BreakGlassActivityFeed.jsx` — live read activity
- `backend/routes/break-glass.js` — Express endpoints

### 6.4 RBAC decision tree (abbreviated)

```
Request arrives
  │
  ├── extract JWT, validate, get {actor, tenant_id, role, break_glass?}
  │
  ├── tenant middleware (runtime/core/tenant_middleware.py, backend/tenancy.js)
  │     enforces actor.tenant_id == resource.tenant_id (unless super_admin context)
  │
  ├── OPA evaluate(actor, resource, action, context)
  │     │
  │     ├── action == 'read_content'?
  │     │     ├── actor.role == 'user' AND owner ─→ ALLOW
  │     │     ├── actor.break_glass == true AND token valid ─→ ALLOW (audit)
  │     │     └── ELSE ─→ DENY (audit)
  │     │
  │     └── action != 'read_content'
  │           └── role permission table lookup
  │
  └── on ALLOW: execute; on DENY: 403 + audit_log entry
```

## 7. Backup + Disaster Recovery

### 7.1 Encryption at backup-time

All backups are encrypted with a **Backup Encryption Key (BEK)** that is *not* derived from any production KEK. Key separation principle: a production-key compromise must not also expose backups.

- **BEK storage:** offline HSM (SaaS) or separate Vault namespace `vault-backup/` with quorum unseal (3-of-5 shares, distributed to officers).
- **Backup target:** S3 with object lock in compliance mode (or equivalent immutable storage), 7-year retention for audit-relevant data, 90-day rolling for ops data.
- **Cipher:** AES-256-GCM, 4 MiB chunked frames with per-chunk IVs.

### 7.2 Postgres

- WAL archiving to S3 with continuous archive
- Base backups daily, encrypted with BEK
- PITR achievable to any second within retention window
- Cross-region replication for `aiemp_ops` (SaaS only)

### 7.3 Chroma

- `scripts/backup_chroma.sh` per-tenant export
- One archive per tenant: `chroma-<tenant_id>-<ts>.tar.gz.enc`
- Encrypted with BEK before upload
- Restore script verifies archive HMAC before decrypting

### 7.4 Vault

- Raft snapshots every 4h, encrypted with BEK
- Stored in separate bucket from app backups (defense in depth)
- Restoration drill quarterly: snapshot → fresh cluster → unseal with archived shards → verify

### 7.5 Restore drill (quarterly)

1. Spin up isolated VPC (no prod network reachability)
2. Restore latest Postgres backup
3. Restore Vault snapshot, unseal in air-gap
4. Decrypt sample row from each major table; verify content is intact
5. Run tenant isolation check: pick 3 random tenants, verify no cross-tenant decryption succeeds
6. Document deltas in `runbooks/quarterly_restore_drill.md`

### 9.3 GDPR

- **Article 17 (Erasure):** `DELETE /api/me/account` triggers a *cryptographic erasure* flow: (1) drop the Chroma per-tenant collection, (2) call `vault transit delete-key kek-<tenant_id>`, (3) mark tenant `crypto_erased_at = now()`, (4) async cascade delete rows over 30 days. After step 2, all ciphertext for that tenant is undecryptable forever, even by us, even from backups.

## Appendix B — Open architectural questions

These need product / legal decisions before implementation:

1. **Default user-non-response policy in break-glass** — auto-approve after 24h vs auto-deny. Default proposed: auto-deny for HIPAA tenants, auto-approve for regular tier.
2. **Vault Enterprise vs OSS** — performance standby on hot data paths is an Enterprise feature; OSS works but with read-after-write latency on the leader.
3. **Per-region key residency** — does any tenant require their KEK to live in a specific region (EU data-residency, US-only for HIPAA BAA covered entities)?
4. **Audit log retention** — minimum 6 years (HIPAA) vs 7 years (SOC2 common). Pick the maximum (7).
5. **Crypto-erasure vs hard delete** — both are GDPR-compliant. Crypto-erasure is faster + works against backups. Some auditors prefer hard-delete-verifiable. Pick one and commit.
